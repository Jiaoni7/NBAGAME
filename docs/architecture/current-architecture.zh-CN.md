# 当前架构说明

这份文档只解释当前项目“现在是怎么组织的”，以及新增公开文件、结构调整、平台扩展时应如何判断放置位置。

## 1. 当前运行模型

当前项目仍然是一个以原生前端文件为核心的单页网页游戏：

- `index.html`：页面结构、样式入口、运行时挂载点
- `game.js`：主要游戏逻辑
- `manifest.webmanifest`：PWA 清单
- `sw.js`：Service Worker 与缓存逻辑
- `scripts/copy-www.js`：把当前 Web 资源复制到 `www/`，供 Capacitor 使用
- `capacitor.config.json`：Android / Capacitor 打包配置

这意味着当前仓库还没有进入需要强行拆分成大型前端工程目录的阶段。

## 2. 目录职责

### 仓库根目录

保留当前运行核心文件与最重要的公开入口：

- `README.md`
- `DEVELOPMENT.md`
- `CONTRIBUTING.md`
- `index.html`
- `game.js`
- `manifest.webmanifest`
- `sw.js`
- `package.json`
- `capacitor.config.json`

这些文件现在留在根目录是合理的，因为它们仍然是当前运行模型的稳定核心，而不是“结构没整理好”。

### `assets/`

提交到仓库、会被运行时直接引用的公开资源。

当前用途：

- `assets/icons/`：图标资源

### `scripts/`

可复用的公开辅助脚本。

当前用途：

- `scripts/copy-www.js`

新增脚本时，优先放这里，而不是散落在根目录。

### `docs/`

稳定的公开文档目录。

当前子分类职责：

- `docs/architecture/`：当前结构说明
- `docs/platforms/`：平台打包与平台差异说明
- `docs/iterations/`：阶段性公开迭代记录
- `docs/audits/`：阶段性审计记录
- `docs/superpowers/specs/`：设计说明
- `docs/superpowers/plans/`：执行计划

### `www/`

由 `npm run www` 生成的 Web 打包产物目录，不作为手工源码维护位置。

### `node_modules/`

依赖安装目录，不作为结构整理目标。

## 3. 新内容该放哪里

默认判断顺序：

1. 是否属于公开文档
2. 是否属于运行时资源
3. 是否属于可复用脚本
4. 是否只是本地私有辅助材料

对应规则：

- 公开稳定文档：放 `docs/` 的既有分类中
- 运行时资源：放 `assets/`
- 可复用脚本：放 `scripts/`
- 本地技能、提示词、研究笔记、内部思路：放被忽略的本地目录，不进仓库

如果新增内容无法自然归入现有分类，先不要立刻创建新的顶层目录；优先写一个简短 spec 或 plan，说明为什么现有结构不够用。

## 4. 为什么当前不急着继续重构目录

当前项目的复杂度主要还在玩法逻辑和上线风险，而不是目录层数不够多。

因此当前阶段的原则是：

- 先保持运行入口稳定
- 先把文档入口和协作规则理顺
- 只有当运行复杂度真的上升时，再做新的结构拆分

换句话说，目录调整应该服务于可维护性，而不是服务于“看起来像大项目”。

## 5. 后续可能的演进方向

如果后面出现下面这些变化，再考虑下一轮结构升级：

- `game.js` 复杂度持续上升，需要模块化拆分
- 需要真实后端、账号、支付或联网能力
- 平台差异明显增多，需要扩展 `docs/platforms/`
- 需要新增多个长期维护的公共资源目录

那时再通过 `docs/superpowers/specs/` 先写设计，确认是否引入新的 `app/`、`src/` 或其他模块目录。
