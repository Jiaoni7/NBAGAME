# 协作文档与结构入口整理设计

日期：2026-06-03

## 背景

仓库已经具备基础的公开文档、状态记录和 Git 协作说明，但当前仍存在几个问题：

- 部分公开文档口吻绑定具体个人角色，不适合长期协作或公开阅读
- 文档入口存在，但“先看什么、改什么、加到哪里”还不够集中
- 缺少一份克制的当前架构说明，导致新增文件时容易靠感觉判断

## 目标

- 让公开文档统一为团队中立视角
- 让人和 AI 都能快速定位项目入口、当前状态、协作流程和结构边界
- 保持小团队协作优先，同时兼容外部贡献者
- 不额外堆出新的文档类别

## 核心决策

### 1. 保留现有文档体系，不新增额外大类

继续沿用现有的：

- `README.md`
- `DEVELOPMENT.md`
- `CONTRIBUTING.md`
- `docs/README.zh-CN.md`
- `docs/project-status.zh-CN.md`
- `docs/changelog.zh-CN.md`
- `docs/git-workflow.zh-CN.md`
- `docs/release-checklist.md`
- `docs/superpowers/specs/`
- `docs/superpowers/plans/`

### 2. 新增一份单点架构说明

增加：

- `docs/architecture/current-architecture.zh-CN.md`

只说明当前运行模型、目录职责、文件放置原则和后续演进边界，不扩展成大型架构手册。

### 3. 统一公开文档为团队中立口吻

避免在公开协作文档中使用“我 / 你 / 朋友”等强绑定表述，改为：

- 个人 fork
- 上游仓库
- 开发分支
- 团队成员
- 外部贡献者

### 4. 明确新工作的决策规则

- 小改动：可直接实施，但结构、命令、流程变化要同步文档
- 中大改动：优先写 spec 或 plan
- 私有技能、提示词、内部思路：只保留在本地忽略目录

## 不采用的方案

### 1. 只改措辞，不调整入口

问题：能缓解口吻问题，但不能解决“先看什么、结构说明在哪”的路由问题。

### 2. 再补更多总览类文档

问题：会增加维护成本，降低文档信号密度，也不符合当前阶段“少而准”的原则。

### 3. 直接上企业级流程文档体系

问题：对当前规模过重，会让维护成本高于实际收益。

## 预期结果

- 新人或新的 AI 对话进入仓库后，能快速找到正确入口
- 团队协作文档不再依赖特定个人视角
- 新增公开内容时，能先判断归属，再决定是否真的需要扩结构
